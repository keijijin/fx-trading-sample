# AI駆動開発における Cursorルール・設計標準・プロンプト運用ノウハウ集

## 1. 目的

本資料は、AIを前提にシステムを構築する際に必要となる実務ノウハウを、次の観点で一体的に整理したものである。

- Cursor ルール記述の考え方
- 設計標準の置き方
- コーディングルールの整理方法
- プロンプトおよび指示リクエストの設計
- プロジェクト運営、レビュー、監査の実務ノウハウ

単に「AIにどう指示するか」ではなく、**AIが継続的に正しい方向へ寄るように、ルール、文脈、標準、運用をどう設計するか** を扱う。

---

## 2. 基本原則

AI駆動開発でまず押さえるべき原則は、次の5点である。

1. AI出力は一次成果物であり、最終成果物ではない
2. 設計、承認、受入、監査可能性の責任は人間が持つ
3. 良い出力は、良いモデルより先に、良い文脈と良い制約から生まれる
4. ルールは「長い総論」より「短く具体的な行動規範」の方が効く
5. AI利用は、ツール導入ではなく、ワークフローと責任分担の再設計である

## 3. Cursor ルール設計の考え方

### 3.1 Cursor ルールで扱うべきもの

Cursor ルールは、毎回プロンプトに書かなくてもよい**恒常的な判断基準**を置く場所である。  
向いているものは次のとおり。

- アーキテクチャ原則
- 禁止事項
- 層ごとの責務分離
- 言語・フレームワークごとのコーディング標準
- ファイル種別ごとの書き方
- レビュー時の必須観点

逆に、向いていないものは次のとおり。

- 一度きりの作業指示
- その場限りの背景説明
- 長すぎる総論や経営メッセージ
- 実行順序を毎回変える手順

### 3.2 良いルールの特徴

- 短い
- 具体的
- 判定可能
- 禁止事項が明確
- 例外が少ない
- できれば「良い例 / 悪い例」がある

### 3.3 悪いルールの特徴

- 理念だけで終わっている
- 複数の論点が混ざっている
- 実務上どう行動すべきかが不明
- 何を禁止したいのかが曖昧
- 長すぎて読み込まれにくい

### 3.4 ルールの分け方

ルールは、1ファイル1テーマが基本である。  
たとえば、次のように分けると扱いやすい。

- `architecture-standards.mdc`
- `java-service-layer.mdc`
- `camel-route-standards.mdc`
- `markdown-docs.mdc`
- `openshift-config.mdc`
- `testing-standards.mdc`

### 3.5 Always Apply と File Specific の使い分け

`alwaysApply: true` に向いているもの:

- プロジェクト全体のアーキテクチャ原則
- セキュリティや監査の禁止事項
- AI利用時の共通ルール

`globs` 指定のルールに向いているもの:

- `**/*.java` の Java コーディング規約
- `**/*.md` の文書記述ルール
- `openshift/**/*.yaml` の設定記述ルール
- `**/*Test.java` のテストルール

### 3.6 ルールの記述例

```markdown
---
description: Camel route responsibilities
globs: backend/**/routes/**/*.java
alwaysApply: false
---

# Camel Route Responsibilities

- RouteBuilder は配線専用にする
- 業務判断、状態遷移、補償判定は Service 層に置く
- 手書き retry / if / loop より Camel EIP を優先する
- 同期呼び出しでサービス境界を崩さない
```

## 4. 設計標準の置き方

### 4.1 設計標準とは何か

設計標準は、個別案件の設計書とは異なり、**そのプロジェクトや組織で繰り返し守るべき設計原則**を定義するものである。  
AI駆動開発では、設計標準があるかどうかで出力品質が大きく変わる。

### 4.2 設計標準に入れるべきもの

- 境界設計
- 真実値の源泉
- トランザクション方針
- 非同期 / 同期の使い分け
- 補償方針
- 冪等方針
- ログと追跡キー
- read model と write model の責務

### 4.3 良い設計標準の書き方

良い標準は、次の書き方になる。

- 原則
- 理由
- 良い例
- 悪い例
- 例外条件

### 4.4 FXトレーディングでの設計標準例

- 約定コアは単一サービス内 `ACID`
- 後続業務は `Saga + Kafka`
- イベント送信は `Outbox`
- Consumer 側は `eventId` 単位の冪等
- 補償は業務的打消し
- `tradeId`、`orderId`、`correlationId`、`eventId` を必須ログ項目とする
- Camel ルートは配線専用

### 4.5 設計標準がないと起きること

- AIが便利そうな同期連携を提案する
- read model を真実値として扱う
- 補償や異常系が曖昧になる
- 設計理由が案件ごとにぶれる
- レビュー基準が揃わない

## 5. コーディングルールの設計

### 5.1 AI向きのコーディングルールとは

コーディングルールは、人間向けの美意識ではなく、**AIが迷いにくい行動指示**として書くと効果的である。

悪い例:

- 可読性を高く保つこと
- 適切に設計すること

良い例:

- 例外は握りつぶさず、ログと再送方針を明記する
- 金額計算に浮動小数点を使わない
- 状態値とイベント名は定数または列挙へ集約する
- 新しい Saga ステップを追加したら、成功 / 失敗 / 補償 / テストを同時に更新する

### 5.2 コーディングルールに入れるべきもの

- 命名
- エラー処理
- ログ出力
- 例外の扱い
- 定数 / 列挙の扱い
- テスト追加方針
- 禁止 API や禁止実装パターン

### 5.3 実装ルールの例

```text
- 金額計算に double / float を使わない
- catch した例外を握りつぶさない
- ログには tradeId, orderId, correlationId, eventId を含める
- 新しいイベントを追加したら consumer 側冪等観点も更新する
- RouteBuilder に業務判定を書かない
```

### 5.4 テストルールの例

```text
- 正常系だけでなく、異常系、補償系、再送系を確認する
- AI生成テストは実装追認型になっていないか確認する
- 負荷試験では p95 / p99 / lag / error rate の基準を先に定義する
```

## 6. プロンプトおよび指示リクエスト設計

### 6.1 プロンプト設計の基本

良いプロンプトは、言葉がうまいのではなく、**前提と完了条件が明確**である。

最低限必要な要素:

- 目的
- 背景
- 前提資料
- 制約
- 依頼内容
- 期待する出力形式
- 禁止事項
- 完了条件

### 6.2 基本テンプレート

```text
目的:
今回達成したいこと

背景:
なぜ必要か

前提:
参照すべき資料、既存方針、制約

依頼内容:
何を比較するか、何を作るか、何をレビューするか

期待する出力:
設計メモ、表、ドラフト、チェックリストなど

禁止事項:
やってはいけないこと

完了条件:
どの状態なら受け入れられるか
```

### 6.3 設計依頼のコツ

- `source of truth` を最初に置く
- 対象と対象外を明確にする
- 段階導入か最終形かを分ける
- 一括変更禁止なら明記する

### 6.4 実装依頼のコツ

- 対象ファイルを絞る
- 既存パターンの踏襲を指示する
- 小さな差分単位で依頼する
- 「壊れそうな点」も併せて聞く

### 6.5 レビュー依頼のコツ

- 何を見てほしいかを先に指定する
- 重大 / 中 / 軽微 で分類させる
- 仕様違反、境界違反、補償漏れ、セキュリティ漏れを分けて見させる

### 6.6 悪い依頼の典型例

```text
良い感じに直して
安全にして
性能を改善して
設計を見て
```

これらはすべて、目的、前提、対象、禁止事項、完了条件が欠けている。

## 7. AIへの指示を強くするための文脈設計

### 7.1 Context Engineering の基本

AIの出力品質を最も左右するのは、モデル性能そのものより、**何を見せるか** である。

重要な観点:

- 必要な資料だけ渡す
- 関係ない履歴を減らす
- 複数テーマを混ぜない
- ルール、標準、仕様を先に渡す
- 実装前に設計制約を固定する

### 7.2 文脈の優先順位

1. システムの設計原則
2. 今回の作業目的
3. 対象ファイルや対象範囲
4. 既存実装パターン
5. 出力形式

### 7.3 具体的に渡すと良いもの

- アーキテクチャ標準
- 設計メモ
- 既存ルールファイル
- 変更対象のファイル
- 関連する試験結果
- 受入条件

### 7.4 渡さない方が良いもの

- 今回関係ない長文資料
- 古い議論ログ
- 機密情報を含む生ログ
- 判断の邪魔になる未整理メモ

## 8. システム構築ノウハウ

### 8.1 最初に整えるべきもの

AIを使ってシステムを作るとき、最初に整えるべきものは次の4つである。

1. 設計標準
2. コーディングルール
3. AI依頼テンプレート
4. レビュー観点テンプレート

この4つがないと、AI利用は個人技になりやすい。

### 8.2 段階導入の考え方

いきなり高リスク領域へ AI を深く入れない。  
順番としては次が現実的である。

1. 文書初稿
2. 既存コード理解
3. テスト観点洗い出し
4. 定型実装
5. 設計比較
6. 高リスク実装の補助

### 8.3 高リスク領域の扱い

次は、AIを使っても最終判断を強く人間が握るべきである。

- 業務真実値に関わるロジック
- 金額計算
- 権限設計
- セキュリティ設定
- DB スキーマ変更
- 本番変更
- 補償ロジック

### 8.4 レビュー体制

AI利用プロジェクトでは、通常レビューに加えて次を揃えると強い。

- 設計レビュー
- 実装レビュー
- AI利用レビュー
- セキュリティレビュー
- リリース判定レビュー

### 8.5 監査・証跡

最低限、次は残すべきである。

- 依頼文
- AI出力
- 採否理由
- レビューコメント
- 承認者
- 反映コミット

## 9. アンチパターン

### 9.1 ルール面

- ルールを1ファイルに全部詰め込む
- 抽象論だけで行動が書かれていない
- 禁止事項が曖昧
- 例外条件が多すぎる

### 9.2 設計面

- AI提案を理由なく採用する
- `source of truth` を明文化しない
- read model を判定に使う
- 便利そうだから同期呼び出しを足す

### 9.3 実装面

- AI差分を読まずにマージする
- テストがあれば安全だと思う
- 例外系や補償系を後回しにする

### 9.4 運用面

- AI出力を無検証で本番へ流す
- 顧客情報や障害ログを無加工で入力する
- 法人向けプランだから安全だと思い込む

## 11. このプロジェクト文脈での適用例

`fx-trading-sample` のようなプロジェクトでは、少なくとも次をルール化しておく価値が高い。

- 約定コアは単一サービス内 `ACID`
- 後続は `Saga + Kafka`
- `Outbox` を前提とする
- Consumer 側は `eventId` 冪等
- 補償は業務的打消し
- Camel ルートは配線専用
- `tradeId`、`orderId`、`correlationId`、`eventId` をログへ残す
- `CQRS` は read path 分離に限定
- `CDC` は `shadow topic` で段階導入
- `Shard` は同期またぎ書き込み禁止

## 12. 結論

システムを AI で構築するときに本当に重要なのは、優れた一発プロンプトではない。  
より重要なのは、**ルール、設計標準、コーディング規約、依頼テンプレート、レビュー運用、監査証跡** を一体として設計することである。

Cursor ルールはその中の一部であり、単独では機能しない。  
ルールが効くのは、設計標準があり、コーディングルールがあり、依頼テンプレートがあり、レビューで受け止める体制があるときである。

したがって、AI駆動開発のノウハウは「プロンプトのうまさ」ではなく、**継続的に AI を正しい方向へ導く情報環境の設計力** にある。
