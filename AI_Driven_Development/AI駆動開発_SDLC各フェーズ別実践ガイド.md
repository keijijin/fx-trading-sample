# AI駆動開発における SDLC 各フェーズ別実践ガイド

## 1. 目的

本資料は、ソフトウェア開発ライフサイクルの各フェーズごとに、生成AIおよびAIエージェントをどう使うと効果的かを整理した実務ガイドである。  
単なる活用例の列挙ではなく、各フェーズにおける `実例`、`勘所`、`ノウハウ`、`注意点`、`やってはいけないこと` をまとめる。

前提とする考え方は次のとおりである。

- AI駆動開発は、コード補完の延長ではなく、SDLC 全体の再設計である
- AI出力は一次成果物であり、最終成果物ではない
- 人間は、判断、受入、承認、監査可能性の責任を持ち続ける
- AIの価値は「雑に速く作ること」ではなく、「論点整理、初稿化、抜け漏れ検出、比較検討の加速」にある

---

## 2. 全体観

AIは SDLC 全体で活用できるが、**どの工程でも同じ使い方が有効なわけではない**。  
実務上は、次のように整理すると理解しやすい。

| フェーズ | AIの主な役割 | 人間の主責任 |
|---|---|---|
| 企画・リサーチ | 調査整理、比較、論点抽出 | 根拠確認、意思決定 |
| 要件定義 | 要件の構造化、漏れ出し、観点整理 | 要件承認、優先順位付け |
| 設計 | 選択肢の比較、設計初稿、制約整理 | 境界設計、採否判断 |
| 実装 | 差分生成、たたき台作成、補助実装 | コードレビュー、受入判断 |
| テスト | 観点生成、ケース作成、データ準備補助 | 戦略設計、重要ケース判断 |
| レビュー・品質保証 | 観点拡張、初期レビュー、説明補助 | 最終判定、トリアージ |
| リリース準備・デプロイ | 手順書初稿、変更影響整理、確認項目整理 | 承認、実行判断 |
| 運用・監視 | 初動整理、ログ要約、仮説生成 | 実行判断、障害対応責任 |

---

## 3. 企画・リサーチ

### 3.1 AI活用の実例

- 市場要求、顧客要求、業界事例の比較整理
- 技術選定候補の長所短所の初期比較
- PoC の論点整理
- 既存課題の言語化

### 3.2 勘所

企画・リサーチ段階では、AIは「答えを出す装置」ではなく、**情報の整理と視点の拡張装置**として使うのが有効である。  
特に、論点を広げる初速や、比較軸の仮置き、曖昧な課題の言語化に向いている。

### 3.3 ノウハウ

- 最初に「何を決めたいのか」を一文で置く
- 比較対象、評価軸、除外対象を明記する
- 根拠の確認が必要な箇所を別欄で出させる
- 事実、仮説、推奨を分けて出力させる

### 3.4 注意点

- AIは流暢に誤るため、調査結果をそのまま事実扱いしない
- 特定ベンダーや技術へのバイアスが出ることがある
- 「比較表がきれい」でも、前提がずれていれば無意味

### 3.5 やってはいけないこと

- 調査結果の出典確認を省略する
- 比較軸を渡さず「おすすめを出して」とだけ依頼する
- 意思決定そのものを AI に委譲する

## 4. 要件定義

### 4.1 AI活用の実例

- ユーザーストーリー、ユースケース、業務フローの初稿作成
- 非機能要件の観点出し
- 例外系、障害系、監査系の抜け漏れ確認
- API 要件、イベント要件、データ要件の構造化

### 4.2 勘所

要件定義で重要なのは、AIに「要件を書かせる」ことではなく、**曖昧さを露出させること**である。  
AIを使うと、要件の穴や暗黙前提が見えやすくなる。したがって、ここでは正解生成よりも、未確定事項の明確化が価値になる。

### 4.3 ノウハウ

- 目的、対象ユーザー、対象業務、除外範囲を最初に書く
- 通常系だけでなく、異常系、遅延、再送、重複、取消も観点に含める
- 「この要件で不足している情報は何か」と聞く
- 承認が必要な要件と、検討中の要件を分ける

### 4.4 注意点

- AIは書きぶりを整えてしまうため、曖昧な要件が一見もっともらしく見える
- 非機能要件や運用要件は、明示しないと落ちやすい
- eventually consistent な領域を、強整合前提で書いてしまうことがある

### 4.5 やってはいけないこと

- 業務真実値と read model を混同した要件を受け入れる
- 規制、監査、保持義務を無指定で AI に任せる
- 仕様未確定のまま詳細実装へ進む

## 5. 設計

### 5.1 AI活用の実例

- アーキテクチャ選択肢の比較
- API / イベント / データモデルの初稿整理
- 責務分離、境界、禁止事項の文書化
- ADR、設計メモ、移行計画、切戻し観点の初稿作成

### 5.2 勘所

設計での AI 活用は、**論点を広げる段階では有効だが、境界を決める段階では人間が強く握る必要がある**。  
AIは便利さを優先しがちで、同期呼び出しの追加、責務の混在、真実値境界の崩壊を招く提案をしやすい。

### 5.3 ノウハウ

- 設計原則と禁止事項を先に渡す
- `source of truth` を最初に固定する
- 選択肢ごとにメリット、デメリット、採用条件、非採用条件を出させる
- 段階導入と最終形を分けて整理させる
- 移行計画では `今やること / 今やらないこと` を必ず分ける

### 5.4 注意点

- AIは「全部やればよい」案を出しやすい
- 構造ボトルネックと読み取り最適化を混同しやすい
- 補償、再送、冪等、順序保証のような難所を軽く扱いがち

### 5.5 やってはいけないこと

- 境界や責務分離を曖昧にしたまま設計案を採用する
- 一括導入前提の設計を、そのまま実行計画にする
- AI提案を理由なく「最新だから」で採用する

### 5.6 FXトレーディングでの実例

- `CQRS` は write path の特効薬ではなく、read path 分離に限定導入する
- `CDC` は本番切替ではなく `shadow topic` で段階検証する
- `Shard` は `accountId` ベースを第一候補にしつつ、複数 shard への同期書き込みは禁止する

## 6. 実装

### 6.1 AI活用の実例

- 定型コード、DTO、マッピング、テスト用ヘルパーの初稿生成
- リファクタリング案の初稿
- 既存コードの読解補助
- 新機能差分のひな形作成

### 6.2 勘所

実装は、AI活用がもっとも成熟している工程である。  
ただし、価値が高いのは「まるごと任せる」ことではなく、**人間が構造を決めた後の差分生成と、既存コード理解の加速**である。

### 6.3 ノウハウ

- 実装前に、対象ファイル、制約、禁止事項を絞る
- 既存設計と同じパターンを踏襲させる
- 1回で完成を狙わず、小さな差分単位で依頼する
- 新規実装と既存コード修正を分けて依頼する
- 「この変更で壊れそうな点」を併せて聞く

### 6.4 注意点

- AIは未使用コードや過剰抽象化を生みやすい
- 既存の命名規則、例外設計、トランザクション境界を破ることがある
- 周辺タブや開いているファイルの情報が入力に含まれるツールもある

### 6.5 やってはいけないこと

- 高リスクの業務ロジックを無差分レビューで受け入れる
- 金額計算、状態遷移、権限、補償判定を AI 任せにする
- 生成差分を読まずにそのままマージする

### 6.6 FXトレーディングでの実例

- `Outbox` relay 最適化のような構造改善では、AIに SQL や更新フローの初稿を作らせても、最終判断は実測ベースで行う
- `Camel RouteBuilder` に業務判定を入れないという制約を先に渡してから変更を依頼する

## 7. テスト

### 7.1 AI活用の実例

- ユニットテスト、結合テストのたたき台生成
- テスト観点、エッジケース、異常系ケースの洗い出し
- テストデータ、入力パターン、モックパターンの案出し
- 負荷試験シナリオや観測指標の整理

### 7.2 勘所

テストでの AI 活用は、**コード生成より観点生成で強い**。  
見落としやすい境界条件、失敗条件、監視項目の洗い出しに向いている。

### 7.3 ノウハウ

- 「何を保証したいテストか」を先に指定する
- 通常系、異常系、再送、重複、遅延、補償を分けて出させる
- カバレッジではなく、業務リスクの高いケースを優先させる
- テストの粒度を指定する

### 7.4 注意点

- AIは実装をなぞるだけの低価値テストを作りやすい
- 見た目のカバレッジが増えても、重要ケースを落とすことがある
- 負荷試験では、測定すべき指標よりシナリオ表現だけが膨らみやすい

### 7.5 やってはいけないこと

- 生成されたテストを実装の正当化に使う
- 異常系や補償系の観点なしで「テストがあるから安心」と判断する
- テスト戦略を決めずにテストコードだけ量産する

### 7.6 FXトレーディングでの実例

- `CDC shadow 検証` では、手順そのものより `成功条件 / 未確認事項 / 比較対象` を AI に整理させるのが有効
- `full-and-stress validation` のような検証では、合否判定基準や観測ポイントを先に固定する必要がある

## 8. レビュー・品質保証

### 8.1 AI活用の実例

- PR の初期レビュー
- 抜けている観点の洗い出し
- 差分の説明文や影響範囲の要約
- 設計逸脱、命名不整合、例外処理漏れの検出補助

### 8.2 勘所

AIレビューは、**代替ではなく補助**として使うのが正しい。  
価値が高いのは、観点の広がりと初期スクリーニングであり、最終承認ではない。

### 8.3 ノウハウ

- 「何を見てほしいか」を指定する
- 性能、整合性、セキュリティ、運用性、境界設計を分けて見させる
- 既存設計原則との整合を確認させる
- 指摘事項は `重大 / 中 / 軽微` に分けさせる

### 8.4 注意点

- AIは「書き方」の指摘に寄りやすく、「設計の危険」を見落とすことがある
- 自信ありげでも、重要欠陥を見逃すことがある
- ノイズの多いレビューになりやすい

### 8.5 やってはいけないこと

- AIレビューだけで承認する
- 指摘ゼロを安全の証拠とみなす
- 人間レビューを形式化してしまう

### 8.6 FXトレーディングでの実例

- `source of truth` の誤り
- `Saga` 補償順序の誤り
- `Outbox` と Consumer 冪等の前提崩れ
- Camel ルートへの業務状態混入

これらは、レビューで特に重点確認すべき論点である。

## 9. リリース準備・デプロイ

### 9.1 AI活用の実例

- リリースノート、変更概要、影響範囲の初稿作成
- デプロイ手順、確認観点、戻し方の整理
- 移行計画の比較案作成
- 関係者向け説明文の作成

### 9.2 勘所

リリース準備では、AIは**作業の抜け漏れ防止と説明文整備**に強い。  
ただし、実行判断や承認判断は必ず人間が持つべきである。

### 9.3 ノウハウ

- 変更対象と変更対象外を分ける
- 成功条件、失敗条件、戻し方を必須項目にする
- 依存する順序と停止条件を明記する
- 「今回やること」と「次段でやること」を分離する

### 9.4 注意点

- AIは移行を一括変更前提で組みやすい
- rollback が弱い手順になりやすい
- 実施前提の環境差分を落としやすい

### 9.5 やってはいけないこと

- AI出力の手順を無検証で実行する
- 本番変更を AI 提案だけで承認する
- 実行者と承認者を同一にする

### 9.6 FXトレーディングでの実例

- `CDC` では `shadow topic` 検証と `full cutover` を分離して扱う
- OpenShift 構成変更では、`設定変更 / 動作確認 / 成功条件 / 未確認事項` を別々に記録する

## 10. 運用・監視

### 10.1 AI活用の実例

- 障害時ログの要約
- 初動切り分け仮説の生成
- アラート説明文の整理
- Runbook の初稿作成

### 10.2 勘所

運用フェーズでは、AIは**初動支援**には有効だが、**実行判断には不向き**である。  
誤誘導の影響が即時で大きいため、運用はもっとも慎重な工程である。

### 10.3 ノウハウ

- 事実ログと推測を分けて出させる
- 影響範囲、暫定対応、恒久対応を分ける
- 類似事例の参照条件を明示する
- 機密情報や顧客情報を入力しない

### 10.4 注意点

- AIは相関と因果を混同しやすい
- 一見もっともらしいが危険な復旧策を返すことがある
- ログに機密情報が含まれる場合、入力自体が問題になる

### 10.5 やってはいけないこと

- AI提案のコマンドを本番でそのまま実行する
- 原因不明のまま恒久対策まで確定する
- 顧客データや監査対象ログを無加工で入力する

### 10.6 FXトレーディングでの実例

- `tradeId`、`orderId`、`correlationId`、`eventId` を軸に、時系列とイベント連鎖を整理させる
- ただし、補償の発火可否や顧客影響判断は人間が最終判断する

## 11. 横断的な勘所

### 11.1 良いAI活用に共通すること

- 目的が明確
- 前提資料が限定されている
- 制約と禁止事項が書かれている
- 完了条件がある
- 受入判断者が明確

### 11.2 失敗しやすいパターン

- 曖昧な依頼
- 全面導入前提
- 真実値境界の未定義
- レビュー観点の未指定
- 速度だけで成功判定する

### 11.3 実務で効く依頼テンプレート

```text
目的:
何を決めたいか、何を作りたいか

背景:
なぜ必要か

前提:
参照すべき資料、既存方針、制約

依頼内容:
比較、提案、初稿、レビュー補助など

期待する出力:
文書、表、差分、観点一覧など

禁止事項:
やってはいけないこと

完了条件:
何が書けていれば完了か
```

## 12. フェーズ別の成熟度感

実務感としては、2026年時点の成熟度はおおむね次のように見てよい。

| フェーズ | AI活用の成熟度 | コメント |
|---|---|---|
| 企画・リサーチ | 中 | 論点整理に強いが、根拠確認が必須 |
| 要件定義 | 中 | 要件の穴を見つけやすいが、承認は人間責任 |
| 設計 | 中 | 比較検討に強いが、境界判断は人間が握る |
| 実装 | 高 | もっとも成熟。ただしレビュー前提 |
| テスト | 中〜高 | 観点生成に強い。戦略設計は人間責任 |
| レビュー・品質保証 | 中 | 補助には有効。代替にはならない |
| リリース準備・デプロイ | 低〜中 | 文書化補助には有効。実行判断は危険 |
| 運用・監視 | 低〜中 | 初動支援は有効。自動実行は慎重に制限すべき |

## 13. 結論

ソフトウェア開発ライフサイクルの各フェーズにおける AI 活用は、**どこで使うか** よりも **どう使うか** が重要である。  
AIは、企画では論点を広げ、要件では曖昧さを露出させ、設計では比較を速め、実装では差分生成を助け、テストでは観点を広げ、レビューでは抜け漏れ検出を補助し、運用では初動整理を支援する。

一方で、どの工程でも共通して守るべき原則は同じである。

1. AI出力は一次成果物である
2. 真実値、責務境界、承認権限は人間が握る
3. 高リスク領域では、レビュー、承認、監査証跡を強制する

この3点を外さなければ、AIは SDLC の多くの工程で実務的な価値を発揮する。  
逆に、この3点を外すと、速く作れても、安全には作れない。
