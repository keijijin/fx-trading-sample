# AI駆動開発研修テキスト 達成度診断設問集

## 1. 位置づけ

本資料は、`AI_Driven_Development/AI駆動開発_研修テキスト.md` に対応した達成度診断用の設問集である。  
事前診断、演習中評価、事後診断、採点ルーブリックをまとめている。

講師はこのまま配布・転記して利用できる。

---

## 2. 診断全体設計

### 2.1 目的

診断の目的は、受講者が以下をどの程度獲得したかを確認することである。

- AI駆動開発の概念理解
- AIへ渡す文脈設計力
- AI出力のレビュー力
- 実務へ展開する導入設計力

### 2.2 評価タイミング

| タイミング | 評価対象 | 方法 |
|---|---|---|
| 事前 | 現状把握 | 自己評価 + 記述 |
| 演習中 | 実践力 | 観察 + 成果物評価 |
| 事後 | 定着度 | 記述課題 + 導入計画 |

### 2.3 評価観点

1. 理解
2. 設計
3. 実行
4. レビュー
5. 展開

## 4. 演習中評価

### 4.1 観察用チェックシート

講師は、演習ごとに以下を確認する。

| 観点 | 見るポイント | 評価メモ |
|---|---|---|
| 理解 | 用語を自分の言葉で説明できるか |  |
| 設計 | 目的、前提、制約、完了条件を分けて書けるか |  |
| 実行 | AIへ具体的な依頼文を作れるか |  |
| レビュー | AI出力の問題点を指摘できるか |  |
| 展開 | 自チーム向けに応用できるか |  |

### 4.2 演習課題別評価

#### 演習A: CDC shadow 検証

評価項目:

- 目的を一文で定義できている
- success criteria が具体的
- 今回の確認範囲と次段課題が分離されている
- AI依頼文に構成前提が含まれている
- 変更点と検証点が混同していない

#### 演習B: CQRS 導入判断

評価項目:

- 書き込み問題と読み取り問題を区別できている
- source of truth を明記できている
- eventual consistency を説明できている
- 導入対象と非対象を切り分けられている
- AIの一般論をそのまま採用していない

#### 演習C: シャーディング方針

評価項目:

- 構造変更の発火条件が具体的
- shard key の理由が説明できる
- 禁止事項を書けている
- サービス境界を崩さない案になっている
- AI出力に対する人間の追記が入っている

## 5. 事後診断

### 5.1 事後課題1: AI依頼文作成

#### 出題

以下の課題に対して、AIへ依頼する文を作成せよ。

課題:  
`outbox_event` から `shadow.fx-trade-events` への CDC 経路を、安全側で検証するための作業計画を作りたい。  
OpenShift、Kafka、PostgreSQL、Kafka Connect を前提とし、本番 topic 切替は行わない。

#### 評価ポイント

- 目的が明確
- 背景が書かれている
- 前提が具体的
- 禁止事項がある
- 完了条件が定義されている

### 5.2 事後課題2: AI出力レビュー

#### 出題

以下のAI提案文を読み、不適切または不十分な点を3つ以上指摘せよ。

```text
性能課題があるため、まず CQRS を全面導入し、すべての read/write を分離するべきです。
その後、必要に応じて Event Sourcing も全面採用します。
約定可否判定も read model に寄せることで、コアDBの負荷を下げられます。
```

#### 模範的な指摘観点

- write path のボトルネックと CQRS の目的を混同している
- 全面導入を前提にしており段階導入になっていない
- 約定可否判定を read model へ寄せるのは不適切
- source of truth の扱いが曖昧
- Event Sourcing 全面採用を安易に連結している

### 5.3 事後課題3: 導入計画作成

#### 出題

自チームを想定し、1ヶ月のAI駆動開発パイロット計画を作成せよ。  
以下を必ず含めること。

- 対象工程
- 対象タスク
- 期待効果
- 主なリスク
- 承認フロー
- 評価指標
- 継続 / 停止判断基準

#### 評価ポイント

- スモールスタートになっている
- 高リスク作業を初手にしていない
- 評価指標が速度だけでない
- レビュー責任が明確
- 見直し条件がある

## 6. 採点ルーブリック

### 6.1 レベル定義

| レベル | 定義 |
|---|---|
| 1 | 用語理解や実践方法が曖昧で、単独では適用困難 |
| 2 | 基本概念は理解しているが、依頼設計やレビューが不十分 |
| 3 | 実務で小規模に適用できる水準 |
| 4 | チーム導入を設計し、他者へ展開できる水準 |

### 6.2 観点別ルーブリック

| 観点 | レベル1 | レベル2 | レベル3 | レベル4 |
|---|---|---|---|---|
| 理解 | AI駆動開発を説明できない | 用語は説明できる | 限界と前提まで説明できる | 導入条件まで説明できる |
| 設計 | 依頼文が曖昧 | 目的は書ける | 前提、制約、完了条件を書ける | 文脈の過不足まで調整できる |
| 実行 | AIを使った成果物作成ができない | 単発の依頼はできる | 初稿化に使える | 案を比較して使い分けできる |
| レビュー | 明白な誤りしか見つけられない | 主要な誤りは指摘できる | 設計観点で修正できる | リスクや欠落観点まで補える |
| 展開 | 現場適用の発想がない | 単発利用に留まる | 小規模パイロットを設計できる | 指標と運用ルール込みで展開できる |

## 7. 判定方法

### 7.1 合格基準例

以下を満たした場合、研修達成とみなす。

- 事後課題の総合評価が `レベル3` 以上
- AI依頼文に `目的 / 前提 / 制約 / 完了条件` が含まれる
- AI出力レビューで本質的な問題を3点以上指摘できる
- 導入計画に `評価指標` と `承認フロー` が含まれる

### 7.2 優秀判定例

以下を満たす場合は、チーム展開候補者として評価できる。

- 総合 `レベル4`
- 自組織の制約を踏まえた現実的なパイロット設計ができている
- レビュー観点が品質だけでなく、運用、境界、リスクまで及んでいる

## 8. 講師向け採点メモ

### 8.1 よくある弱点

- AIに何をさせたいかは書けているが、何をさせてはいけないかが書けていない
- 「速くなる」以外の評価指標がない
- AI出力の誤りを文体や表現の問題としてしか捉えられていない
- 自チーム導入時の承認フローが曖昧

### 8.2 高評価の特徴

- 依頼文に現場制約が自然に入っている
- 設計文書の採否条件を説明できる
- AIの出力を起点に、人間が追加で論点を補っている
- チーム運用へ落とした時の現実性が高い

## 10. 研修後の活用

達成度診断は、受講者の合否だけでなく、次の改善にも利用できる。

- どの説明が伝わりにくかったか
- どの演習でつまずきが多かったか
- 組織としてどの観点のガバナンスが弱いか
- 次回研修で補強すべき内容は何か

診断結果は、個人評価に閉じず、チーム導入の設計材料として使うことが望ましい。
