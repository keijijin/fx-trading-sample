# AI駆動開発研修テキスト スライド骨子

## 1. 目的

本資料は、`AI_Driven_Development/AI駆動開発_研修テキスト.md` をもとに、PowerPoint や Google Slides へ落とし込みやすいよう整理したスライド骨子である。  
講義用、社内説明用、キックオフ用のたたき台として利用できる。

想定は、`20枚前後` の標準版である。  
必要に応じて、`10枚版` の短縮説明へ圧縮してもよい。

---

## 2. 全体構成

### Slide 1. タイトル

**タイトル案**  
現場事例で学ぶ AI駆動開発  
`fx-trading-sample` による設計・実装・検証の実践

**サブタイトル案**

- AIを前提にした SDLC 再設計の実務
- コード補完から、設計・検証・運用準備まで

**話すポイント**

- 本研修の対象
- 今日のゴール
- なぜこのテーマが今重要か

### Slide 2. 今日伝えたいこと

**見出し**

今日のメッセージは3つ

**本文案**

1. AI駆動開発は、コード生成ではなく開発様式の再設計である
2. 良い成果は、良い文脈設計と良いレビューから生まれる
3. 導入は、小さく試し、測り、改善しながら広げる

### Slide 3. AI駆動開発とは何か

**見出し**

AI駆動開発の定義

**本文案**

- 生成AIとAIエージェントを SDLC 全体へ組み込む
- 要件定義、設計、実装、レビュー、運用を再設計する
- 重要判断は人間が保持する

**図案**

```text
Human prompts -> Agent executes -> Human reviews
```

### Slide 4. 何がAI駆動開発で、何が違うのか

**見出し**

関連概念の整理

**本文案**

- AI-assisted coding: 点的な支援
- Vibe coding: 速いが本番規律に弱い
- Agentic engineering: 自律実行を含むワークフロー
- Spec-driven / Context Engineering: 品質安定化の鍵

**話すポイント**

- 補完利用だけではプロセス変革にならない
- 本番では「速さ」より「受入責任」が重要

### Slide 5. なぜ今、AI駆動開発なのか

**見出し**

背景にある3つの圧力

**本文案**

- 人材制約
- タイム・トゥ・マーケット圧力
- レガシーと技術的負債

**補足**

- ツール成熟
- マルチエージェント化
- MCP など周辺技術の整備

### Slide 6. 効果は一様ではない

**見出し**

「速くなる」は条件付き

**本文案**

- 小さな新規実装では効きやすい
- 既存の成熟コードベースでは逆に遅くなる場合もある
- 最後の30%は人間の設計・レビューが重い

**話すポイント**

- 体感と実測はズレる
- 導入効果は、ベースラインと指標で測る必要がある

### Slide 7. リスクは4分類で考える

**見出し**

AI利用時の主要リスク

**本文案**

- 品質
- セキュリティ
- 自律性
- コンプライアンス

**話すポイント**

- 危険なのは「自信満々に間違える」こと
- 法人向け利用でも、データフロー確認は必要

### Slide 8. ガバナンスの原則

**見出し**

Bounded Autonomy

**本文案**

- 低リスク: 自律実行可
- 中リスク: 実行後に人間レビュー
- 高リスク: 事前承認必須

**図案**

```text
Low risk -> Medium risk -> High risk
auto       review        approve first
```

### Slide 9. Context Engineering が成否を分ける

**見出し**

良いAI出力は、良い文脈から生まれる

**本文案**

- 何を見せるか
- 何を見せないか
- どの順で渡すか
- どこで完了とするか

**話すポイント**

- プロンプトだけでは不十分
- 参照資料の選別が重要

### Slide 10. 良い依頼文の型

**見出し**

AI依頼テンプレート

**本文案**

- 目的
- 背景
- 前提
- 依頼内容
- 期待する出力
- 禁止事項
- 完了条件

**話すポイント**

- 人間の準備が出力品質を決める

### Slide 11. 今回の研修題材

**見出し**

`fx-trading-sample` の3つのケース

**本文案**

- CDC shadow topic 検証
- CQRS 導入条件の整理
- シャーディング・ドメイン分割ロードマップ

### Slide 12. ケース1: CDC shadow 検証

**見出し**

安全側で検証を進める

**本文案**

- `outbox_event -> WAL -> Debezium -> Kafka Connect -> shadow topic`
- 本番切替ではなく、shadow で経路成立を確認
- 検証と切替を分ける

**学び**

- 段階導入
- 構成変更の記録
- 成功条件と次段課題の分離

### Slide 13. ケース2: CQRS 導入判断

**見出し**

「何に効くか」を間違えない

**本文案**

- CQRS は write path の特効薬ではない
- read model は読み取り分離に効く
- source of truth は ACID core に残す

**学び**

- AIに論点整理をさせる
- 採否条件は人間が決める

### Slide 14. ケース3: シャーディング方針

**見出し**

構造変更は、条件を満たした時だけ行う

**本文案**

- `accountId` ベース shard を第一候補
- shard 間 2PC / XA は禁止
- Camel ルートは配線専用

**学び**

- 境界設計
- 禁止事項の明文化
- 移行フェーズ設計

### Slide 15. 研修で身につける力

**見出し**

受講後にできるようになること

**本文案**

- 依頼前に仕様と制約を整える
- AI出力をレビューする
- 設計判断を文書として残す
- 自チーム向けのパイロットを設計する

### Slide 16. カリキュラム

**見出し**

標準2日版の全体像

**本文案**

- Day 1: 基礎理解、リスク、依頼文設計
- Day 2: ケース演習、導入計画、診断

**補足**

- 1日版、半日版へ圧縮可能

### Slide 17. 達成度診断

**見出し**

知識ではなく、実践力を測る

**本文案**

- 事前診断
- 演習評価
- 事後診断
- 5観点評価: 理解 / 設計 / 実行 / レビュー / 展開

### Slide 18. 導入効果の測り方

**見出し**

速度だけで測らない

**本文案**

- 初稿作成時間
- レビュー差戻し率
- 手戻り件数
- 品質指標
- 開発者満足度

### Slide 19. 導入の進め方

**見出し**

小さく始めて、標準化する

**本文案**

1. 低リスク業務を選ぶ
2. テンプレートを用意する
3. レビュー責任を決める
4. 指標を取る
5. 成功パターンだけを標準化する

### Slide 20. まとめ

**見出し**

AI時代の開発責任を再設計する

**本文案**

- AI駆動開発は、便利ツールの話ではない
- 人間の判断、レビュー、説明責任はむしろ重要になる
- 実践は、現場の文脈に合わせて段階導入すべきである

## 4. デザイン上の注意

- 文章を詰め込みすぎず、1スライド1メッセージにする
- 図は詳細アーキテクチャより、責任分担と流れを見せる
- ツール名比較は主役にしない
- ケーススライドでは「技術詳細」より「判断ポイント」を前に出す
