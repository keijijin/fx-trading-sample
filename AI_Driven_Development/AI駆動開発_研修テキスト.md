# AI駆動開発研修テキスト

## 1. 本テキストの位置づけ

本テキストは、`fx-trading-sample` を題材にした今回の開発体験をもとに、AI駆動開発を実務へ適用するための研修教材として作成したものである。  
参照元となる `AI_Driven_Development/AI駆動開発_実践概説書.md` が「AI駆動開発とは何か」「なぜ今必要か」「どのようなリスクと設計原則があるか」を整理した概説編であるのに対し、本テキストは「実際の開発プロジェクトでどう運用し、何を学習し、どう評価するか」を扱う実践編である。

本研修は、単に生成AIでコードを書く方法を教えるものではない。目的は、生成AIとAIエージェントを前提に、要件定義、設計、実装、レビュー、検証、運用準備までを再設計する実務力を養うことである。

## 2. 研修企画

### 2.1 研修名

**現場事例で学ぶ AI駆動開発研修  
`fx-trading-sample` による設計・実装・検証の実践**

### 2.2 企画背景

AI駆動開発は、コード補完の延長ではなく、SDLC全体の進め方を再設計するアプローチである。今回の開発では、以下のような実務的テーマに対して、AIを使いながら設計・検証・文書化を進めた。

- `Outbox + Kafka + Saga` を前提としたイベント駆動アーキテクチャの整理
- `CQRS` の適用条件と導入判断の明文化
- `accountId` ベースのシャーディング方針の設計
- `Debezium + Kafka Connect` による CDC shadow topic 検証
- OpenShift / Kafka / PostgreSQL の構成変更と検証記録
- 負荷、整合性、拡張性、運用性を意識した設計ドキュメント化

これらの活動は、AI駆動開発の本質である次の要素を含んでいる。

- AIに渡す前提情報を整える `Spec-driven development`
- 必要な資料だけを読ませる `Context Engineering`
- AIに下書きをさせつつ、人間が判断と受入責任を持つ `Human-prompted -> Agent-executed -> Human-reviewed`
- 出力物を設計書、検証レポート、判断基準として残すガバナンス設計

### 2.3 研修の狙い

受講者が、AI駆動開発を「便利なツール利用」ではなく、**設計・検証・意思決定の質を高める開発様式**として理解し、自チームへ段階導入できるようにすることを狙う。

### 2.4 対象者

- アプリケーション開発者
- テックリード
- アーキテクト
- SRE / プラットフォーム担当
- 開発マネージャ
- AI導入を検討する推進担当

### 2.5 前提スキル

- Git、IDE、レビュー、テストの基本を理解している
- API、RDB、メッセージングの基本を理解している
- Java / Spring Boot / Camel / Kafka のどれかに触れた経験があると望ましい
- 生成AIツールを少なくとも一度は触ったことがあると理解しやすい

### 2.6 研修後に期待する行動変容

受講後、受講者は次の行動を取れる状態を目指す。

1. AIへ丸投げせず、仕様、制約、判断材料を先に整えてから依頼できる
2. AIの出力を、設計・性能・整合性・運用性の観点でレビューできる
3. 開発体験を再現可能なドキュメントと判断ログとして残せる
4. 自チームに合う小さなパイロットを企画し、段階導入できる

## 3. 研修目的

### 3.1 学習目的

本研修の学習目的は、以下の4点である。

1. **AI駆動開発の定義を正しく理解する**  
   AI駆動開発を、単なるコード生成ではなく、SDLC全体とガバナンスの再設計として理解する。

2. **現場で使える実践手順を身につける**  
   要件整理、設計判断、実装補助、レビュー、検証、文書化を、AIと協働しながら進める型を身につける。

3. **リスクを制御しながら成果を出す方法を理解する**  
   ハルシネーション、過剰な自律性、品質劣化、情報漏洩、レビュー不足といったリスクへの対策を理解する。

4. **自組織への導入計画を立てられるようにする**  
   パイロット対象、評価指標、役割分担、教育設計を含めた導入プランを作成できるようにする。

### 3.2 業務目的

本研修は、単なる知識習得ではなく、次の業務成果に結びつけることを目的とする。

- 設計書、ADR、検証レポートの作成速度を上げる
- 論点整理の初速を上げつつ、判断品質は維持する
- レビューと検証の抜け漏れを減らす
- チーム内でAI活用方法のばらつきを抑える
- AI利用時のガバナンスと説明責任を強化する

## 4. この研修で扱う開発体験

本研修では、抽象論だけでなく、実際の開発活動に近い以下の題材を使う。

### 4.1 ケースA: CDC shadow topic 検証

`Debezium + Kafka Connect` を使い、`outbox_event` から `shadow.fx-trade-events` へ CDC 配信できるかを確認した事例を扱う。  
学習の焦点は、次のとおりである。

- AIに調査や作業計画を依頼する際に、何を前提情報として渡すべきか
- OpenShift、Kafka、PostgreSQL の設定変更をどう安全に記録するか
- 「本番切替」ではなく「shadow 検証」に留める段階的導入の考え方
- 技術的成功条件と、次段で確認すべきリスクをどう文書化するか

### 4.2 ケースB: CQRS 導入条件の整理

「CQRSを導入すべきか」という問いに対して、AIに説明文を書かせるだけでなく、適用条件、非適用条件、read model 候補、整合性の説明責務を明文化した事例を扱う。  
ここでは次を学ぶ。

- AIに論点を広げさせた後、人間が適用範囲を狭めていく方法
- 書き込み性能問題と読み取り分離問題を混同しない思考法
- AIの出力を設計判断文書へ昇格させるレビューの観点

### 4.3 ケースC: シャーディング・ドメイン分割ロードマップ

`fx-core-service` が将来ボトルネックになった場合を想定し、`accountId` ベースの shard を第一候補とする設計方針をまとめた事例を扱う。  
学習の焦点は、次のとおりである。

- 性能改善と構造変更をどう切り分けるか
- 分割原則、禁止事項、移行フェーズをAIと共同で整理する方法
- Camel、Saga、Kafka、Outbox を前提とした境界設計
- 「ルートは配線専用、業務判断は Service 層へ」という設計規律の守り方

### 4.4 ケースD: AI活用そのもののふりかえり

上記の成果物を素材として、以下の観点からAI活用をふりかえる。

- どのタスクはAIで速くなったか
- どのタスクはむしろ人間の判断負荷が高かったか
- どの場面で追加文脈が必要だったか
- どの場面でレビュー観点を強める必要があったか

### 6.2 モジュール別詳細

#### モジュール1: AI駆動開発の全体像

**狙い**  
AI駆動開発を「コード自動生成」ではなく「開発様式の再設計」として理解する。

**扱う内容**

- AI駆動開発の定義
- AI-assisted coding、Vibe coding、Agentic engineering の違い
- Human oversight の必要性
- 生産性向上が条件依存であること
- 品質・セキュリティ・自律性・コンプライアンスの4リスク

**演習**

- 自チームの開発活動を `企画 / 要件 / 実装 / テスト / レビュー / 運用` に分け、AIが効きそうな場面と危険な場面を洗い出す

**成果物**

- チーム別 AI活用マップ

#### モジュール2: Context Engineering と Spec-driven Development

**狙い**  
AIに良い出力をさせる前に、人間側が何を整理すべきかを理解する。

**扱う内容**

- 良い依頼と悪い依頼の違い
- 仕様、制約、前提、非機能要求、完了条件の書き方
- 参照資料の選び方
- 文脈の過不足が品質へ与える影響
- 指示、背景、入力、期待出力、禁止事項のテンプレート化

**演習**

- `CQRS 導入条件と適用方針` を題材に、AIへ渡す前提情報を整理する
- 「CQRSを導入すべきか」という曖昧な問いを、判断可能な依頼文へ書き換える

**成果物**

- AI依頼テンプレート
- 設計判断テンプレート

#### モジュール3: ケース演習 1 - CDC shadow 検証

**狙い**  
AIを使った技術検証の計画、実施、記録の残し方を学ぶ。

**扱う内容**

- 成功条件、失敗条件、次段課題の明文化

**演習**

- CDC 検証の作業計画をAIに起草させる

**成果物**

#### モジュール4: ケース演習 2 - CQRS とシャーディングの設計判断

AIを使って論点を広げつつ、人間が設計の境界と採否条件を明確にする練習を行う。

**扱う内容**

- eventual consistency の説明責任

**演習**

- 「性能が悪いからCQRSを入れるべきではないか」という意見に対し、AIを補助役として使いながら反証・整理する
- shard 設計案をAIに整理させ、その中から禁止事項や移行リスクを人間が追記する

**成果物**

- CQRS 導入判断メモ

## 8. 研修で使う基本フレーム

### 8.1 AI依頼テンプレート

以下の型で依頼すると、AIの出力品質が安定しやすい。

```text
目的:
今回達成したいことは何か

背景:
なぜこの作業が必要か

前提:
参照すべきファイル、既存方針、制約条件

依頼内容:
何を作るか、何を比較するか、何を判断するか

期待する出力:
文書、表、設計案、レビュー観点、テスト観点など

禁止事項:
やってはいけないこと、対象外のこと

完了条件:
どの状態になれば作業完了とみなすか
```

### 8.2 AIレビュー観点テンプレート

AI出力を受け入れる前に、最低限次を確認する。

1. 要求された範囲に答えているか
2. 前提資料と矛盾していないか
3. 重要な制約を落としていないか
4. 設計判断の理由が書かれているか
5. 運用、性能、整合性、セキュリティの観点が抜けていないか
6. 次のアクションが判断できる形になっているか

### 8.3 本研修で重視する設計原則

`fx-trading-sample` の題材では、次の原則を守る。

- 約定コアは単一サービス内 ACID を維持する
- 後続処理は Saga + Kafka で分離する
- Camel ルートは配線専用とし、業務判断は Service 層に置く
- イベント送信は Outbox を前提とする
- Consumer 側では冪等制御を必須とする
- 補償は DB ロールバックではなく業務的打消しで表現する
- 同期呼び出しでサービス境界を崩さない

これらの原則をAIへ明示しないと、便利そうでも不適切な案を生成しやすい。  
したがって、AI活用では「自由に考えさせる」前に「守るべき境界を渡す」ことが重要である。

## 9. 達成度診断

### 9.1 診断の考え方

達成度診断は、単なる知識テストではなく、**理解、設計、実行、レビュー、展開** の5観点で行う。  
また、研修前後で比較できるように、事前診断、演習評価、事後診断の3段階で測定する。

### 9.2 事前診断

研修開始前に、以下を自己評価と簡易記述で確認する。

**自己評価項目**

- 生成AIを実務で使ったことがある
- AIに仕様や設計相談をしたことがある
- AI生成コードをレビューしたことがある
- AI利用時の情報管理ルールを説明できる
- AI導入効果を測る指標を考えたことがある

各項目を `1: ほぼできない` から `5: かなりできる` で回答する。

**事前記述問題**

1. AI駆動開発とコード補完の違いを100字程度で説明せよ
2. AIに設計相談をするとき、最低限渡すべき情報を3つ挙げよ
3. AI生成物のレビューで重要だと思う観点を3つ挙げよ

### 9.3 演習中評価

演習では、以下のルーブリックで講師と受講者相互に評価する。

| 観点 | レベル1 | レベル2 | レベル3 | レベル4 |
|---|---|---|---|---|
| 理解 | 用語の説明が曖昧 | 概念は説明できる | 違いと限界を説明できる | 実務上の適用条件まで説明できる |
| 設計 | 指示が曖昧 | 前提は書ける | 制約と完了条件を書ける | 文脈の過不足まで調整できる |
| 実行 | AIへ依頼できない | 依頼できるが粗い | 成果物の初稿を作れる | 複数案を比較して使い分けできる |
| レビュー | 誤りを見逃す | 明白な誤りは見つけられる | 設計・制約の観点で修正できる | リスクや欠落観点まで補える |
| 展開 | 自チームへ適用できない | 単発利用は考えられる | 小規模パイロットを設計できる | 指標と運用ルール込みで展開できる |

### 9.4 事後診断

研修後、次の課題を実施する。

**課題1: 設計依頼文の作成**  
「CDC shadow topic 検証を安全側で進めるための作業計画をAIへ依頼する文」を作成する。

**課題2: レビュー課題**  
AIが出力した CQRS 提案文から、不適切または不十分な点を3つ以上指摘する。

**課題3: 導入計画課題**  
自チームに対して、1ヶ月のAI駆動開発パイロット計画を作成する。

### 9.5 合格基準の例

以下を満たした場合、研修達成とみなす。

- 事後診断で総合 `レベル3` 以上
- 導入計画課題において、対象工程、評価指標、レビュー責任が明記されている
- AI依頼文に、目的、前提、制約、完了条件が含まれている

## 10. 研修運営プラン

### 10.1 実施形態

推奨は、以下のいずれかである。

- 2日間集中ワークショップ
- 半日 x 4回の分割実施
- 講義編 1日 + 演習編 1日

### 10.2 受講人数

- 推奨: 12名から24名
- 演習中心の場合は、4名から6名のグループ編成が望ましい

### 10.3 必要な準備

- `fx-trading-sample` の概要説明資料
- 参照ドキュメント一式
- 利用するAIツールと利用ルール
- 演習用テンプレート
- 診断シート

### 10.4 講師側の準備

講師は、単にAIを使えるだけでなく、次を説明できる必要がある。

- AIを使う前に何を決めるべきか
- AI出力のどこが危険か
- 良い依頼と悪い依頼の差は何か
- 設計判断をどうレビューし、どう文書として固定するか

## 11. 研修後の定着施策

研修で終わらせず、現場定着へつなげるために、以下の施策を推奨する。

### 11.1 1ヶ月パイロット

対象を限定し、次のような低から中リスク領域で試行する。

- 設計メモの初稿作成
- 検証レポートの初稿作成
- テスト観点の洗い出し
- 既存コード理解の補助

### 11.2 レビュー会

週1回、30分から60分で、以下を共有する。

- うまくいった依頼文
- 失敗した依頼文
- AIが見落とした観点
- 追加したガードレール

### 11.3 テンプレート化

研修で作った以下を標準化する。

- AI依頼テンプレート
- 設計判断テンプレート
- レビュー観点チェックリスト
- 検証レポートテンプレート

### 11.4 効果測定

少なくとも次を継続計測する。

- 初稿作成時間
- レビュー差戻し率
- 手戻り件数
- ドキュメント作成数
- 開発者の主観満足度

## 12. まとめ

AI駆動開発の研修で本当に教えるべきことは、「AIに何を頼めるか」だけではない。  
より重要なのは、「AIに何を渡すべきか」「AIの出力をどう検証するか」「どこで人間が責任を持つか」を、実際のプロジェクト文脈で学ぶことである。

`fx-trading-sample` の開発体験は、この点において教材として非常に適している。  
なぜなら、単なるサンプル実装ではなく、以下のような実務の本質が含まれているからである。

- 設計判断に理由が必要であること
- 段階導入が重要であること
- 性能、整合性、運用性を同時に考える必要があること
- AIの出力だけでは不十分で、人間のレビューが不可欠であること

本研修は、AIを使う研修ではなく、**AI時代の開発責任を再設計する研修**として位置づけるべきである。  
この位置づけが明確であれば、受講者は「便利だから使う」段階を超えて、「成果と統制を両立するために使う」段階へ進むことができる。
