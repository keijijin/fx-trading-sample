## 2. 事前記述問題 模範解答例

### 問1

**設問**  
AI駆動開発と、単なるコード補完中心のAI利用との違いを、100字から150字程度で説明せよ。

**模範解答例**

```text
AI駆動開発は、生成AIをコード補完に使うだけでなく、要件定義、設計、実装、レビュー、検証、運用まで含めて進め方を再設計する考え方である。重要な判断や受入責任は人間が担い、AIは一次成果物の生成や論点整理を担う。
```

**評価の観点**

- SDLC 全体に触れているか
- 単なる補完との差が書かれているか
- 人間の責任が残ることに触れているか

**設問**

- 背景
- 参照すべき前提資料
- 制約条件
- 対象外とする事項
- 完了条件

### 問3

**設問**

**模範解答例**

## 3. セッション3 演習 模範解答

### 3.1 悪い依頼の例

```text
CQRS を入れるべきか見てください。よさそうな案をまとめてください。
```

**問題点**

- 何のための検討かが不明
- 現状のボトルネックが不明
- 参照資料が不明
- 出力形式が不明
- 対象外が不明
- 完了条件が不明

### 3.2 良い依頼文の例

```text
目的:
fx-trading-sample において CQRS を導入すべき領域と、現時点で導入しない領域を整理したい。

背景:
読み取り系 API や運用ダッシュボード要求の増加を想定しているが、現在の主ボトルネックは約定 ACID コア、Outbox relay、Kafka lag、Saga 状態集中である。

前提:
design/cqrs-introduction-criteria.md を主資料として参照すること。
現行の source of truth は ACID core であり、CQRS は write path 改善の第一優先ではない。

依頼内容:
CQRS の導入目的、導入対象、導入しない対象、read model 候補、eventual consistency の説明事項を整理してほしい。

期待する出力:
見出し付きの設計メモ形式。導入してよい対象と避ける対象を分け、理由を添えること。

禁止事項:
CQRS を全面導入前提で書かないこと。約定可否判定や残高の真実値を read model に寄せないこと。

完了条件:
source of truth、適用範囲、非適用範囲、整合性説明が明記された設計メモになっていること。
```

**評価の観点**

- 主資料が特定されている
- ボトルネックが先に書かれている
- 禁止事項が明確
- AIが一般論へ逃げにくい

## 4. ケース演習 1: CDC shadow 検証 模範解答

### 4.1 目的整理

**模範解答例**

```text
outbox_event から Kafka の shadow topic まで、Debezium と Kafka Connect を経由した CDC 経路が OpenShift 上で成立するかを、安全側の構成で確認する。
```

### 4.2 前提情報

**模範解答例**

- 対象は `outbox_event` から `shadow.fx-trade-events` までの経路である
- 本番 topic への切替は行わない
- OpenShift 上で Kafka Connect と PostgreSQL logical decoding を使う
- 確認対象は connector 状態と shadow topic への実データ到達である

### 4.3 成功条件

**模範解答例**

- connector が `RUNNING` である
- task が `RUNNING` である
- `POST /api/trades` 実行後に shadow topic からイベントを受信できる
- payload に `tradeId`、`orderId`、`correlationId` など主要項目が含まれる

### 4.4 今回まだ確認していないこと

**模範解答例**

- poller publish との payload 差分
- 順序保証の妥当性
- 欠落がないことの継続確認
- full cutover 時の影響

### 4.5 AIへの依頼文 例

```text
目的:
outbox_event から shadow.fx-trade-events への CDC 経路を、安全側で検証する作業計画を作成したい。

背景:
本番 topic 切替前に、PostgreSQL WAL、Debezium、Kafka Connect を経由した配信成立を確認したい。

前提:
OpenShift 上で fx-postgres、Kafka 3 broker、Kafka Connect を利用する。
対象は shadow topic への配信成立確認であり、本番 topic 切替や poller 停止は今回行わない。

依頼内容:
事前確認、設定変更、動作確認、結果記録、次段課題整理の順で、実施計画を作成してほしい。

期待する出力:
手順書形式の計画書。成功条件と今回未実施項目を分けて記述すること。

禁止事項:
本番切替を前提にしないこと。poller 停止や full cutover を今回の完了条件に含めないこと。

完了条件:
変更点、検証手順、成功条件、次段課題が分離された計画書になっていること。
```

### 4.6 AI出力レビューの模範指摘

- 検証計画なのに、本番切替まで含めていたら過剰
- 設定変更の羅列だけで、成功条件が書かれていなければ不十分
- 今回確認する範囲と、次段で比較すべき事項が混ざっていれば修正対象

### 4.7 学びの例

- AIで速くなった点: 作業計画のたたき台整理
- 人間判断が必要だった点: 本番切替を含めない範囲設定
- 次回改善したい点: 成功条件と観測方法を最初から明文化する

## 5. ケース演習 2: CQRS 導入判断 模範解答

### 5.1 論点整理

**模範解答例**

- 現状の主ボトルネックは write path 側にある
- 一方で、trace、e2e-status、ダッシュボード系の read 要求は今後増える可能性がある
- したがって、CQRS は write path 改善ではなく read 分離目的で限定導入を検討すべきである

### 5.2 CQRS 導入の目的

**模範解答例**

- 読み取り API と書き込み API の責務分離
- core DB への read 負荷軽減
- 集約済み read model による照会高速化

### 5.3 導入してよい対象

**模範解答例**

- `GET /api/trades/{tradeId}/trace`
- `GET /api/trades/{tradeId}/e2e-status`
- 運用ダッシュボード
- 監査向け統合照会

### 5.4 導入しない対象

**模範解答例**

- 約定可否判定
- 残高拘束の真実値
- 建玉の ACID 主データ

### 5.5 eventual consistency の説明文

```text
本 read model は source of truth ではなく、integration event に基づく投影結果である。そのため、write 直後の短時間は最新状態を反映しない可能性がある。特に trace や e2e-status は eventually consistent であり、真実値が必要な判定は引き続き ACID core 側で行う。
```

### 5.6 AI出力レビューの模範指摘

- 「性能課題だから全面CQRS」は論点のすり替え
- source of truth が明記されていないなら不十分
- read model に寄せる対象と寄せない対象の境界が必要

## 6. ケース演習 3: シャーディング方針 模範解答

### 6.1 構造変更の発火条件

**模範解答例**

- レプリカを増やしても `POST 201 rate` が頭打ちになる
- DB 接続予算が継続的に逼迫する
- `accountId` 単位の更新競合が支配的になる

### 6.2 shard key の候補整理

| 候補 | メリット | デメリット | 判断 |
|---|---|---|---|
| `accountId` | 残高拘束、建玉更新、約定記録の整合主体と一致しやすい | 跨口座要件に弱い | 第一候補 |
| 地域 / 法人 | 規制や運用境界に合わせやすい | 性能境界と一致しない場合がある | 第二候補 |
| 商品 / ドメイン | 業務責務の分離に有効 | 現行の主競合点と直結しにくい | 将来候補 |

### 6.3 守るべき原則

- 約定コアは shard 内で強整合
- 後続は Kafka + Saga による最終的整合
- Outbox による送信保証
- Camel ルートは配線専用
- shard 判定と業務判断は Service / resolver 層

### 6.4 禁止事項

- 同一リクエストで複数 shard へ同期書き込みしない
- shard 間 2PC / XA を導入しない
- shard またぎの強整合参照を下流へ要求しない

### 6.5 AI出力に人間が追記すべき観点

- 再配置時の運用手順
- ロールバック条件
- ログ、trace、監視に `shardId` をどう出すか

## 7. 事後課題 模範解答

### 7.1 事後課題1: AI依頼文作成

**模範解答例**

```text
目的:
outbox_event から shadow.fx-trade-events への CDC 経路が、安全側の構成で成立するかを確認する作業計画を作成したい。

背景:
本番 topic 切替前に、PostgreSQL WAL、Debezium、Kafka Connect を経由した配信成立を確認し、切替判断の前提情報を得たい。

前提:
OpenShift 上に fx-postgres、Kafka 3 broker、Kafka Connect を配置する。
対象は shadow topic への到達確認であり、本番 topic 切替、poller 停止、full cutover は今回行わない。

依頼内容:
事前確認項目、必要設定、実施手順、成功条件、結果記録方法、次段課題を整理してほしい。

期待する出力:
手順書形式の作業計画。今回の確認範囲と、将来の切替試験で確認すべき範囲を分けて記述すること。

禁止事項:
本番切替を完了条件に含めないこと。今回の範囲外の作業を成功条件に混ぜないこと。

完了条件:
変更点、確認手順、成功条件、未確認事項が明確に分かれた作業計画になっていること。
```

### 7.2 事後課題2: AI出力レビュー

**模範解答例**

1. write path の性能課題に対して、CQRS を全面導入すべきと短絡しており、適用目的を混同している
2. 全 read/write 分離を前提にしており、段階導入の観点が欠けている
3. 約定可否判定を read model へ寄せる提案は、ACID 側の真実値と責務境界を崩すため不適切
4. source of truth が明記されていない
5. Event Sourcing の全面採用を追加前提のように扱っており、設計判断が粗い

### 7.3 事後課題3: 導入計画作成

**模範解答例**

```text
対象工程:
設計とドキュメント作成

対象タスク:
設計メモ、検証レポート、レビュー観点整理の初稿作成

期待効果:
初稿作成時間の短縮、論点漏れの削減、設計理由の明文化

主なリスク:
一般論に流れる、レビュー負荷増、誤った前提のまま文章化される

承認フロー:
AIが初稿作成 -> 担当者レビュー -> テックリード承認 -> 共有

評価指標:
初稿作成時間、レビュー差戻し率、手戻り件数、利用者満足度

継続判断基準:
初稿作成時間が短縮し、レビュー差戻し率が許容範囲内であること

停止判断基準:
レビュー負荷増により全体工数が改善しない、または誤記が継続すること
```

## 8. 講師向け運用メモ

### 8.1 模範解答の使い方

- 受講者の回答が模範解答と完全一致する必要はない
- 重要なのは、観点が揃っているかどうか
- 用語の正確さだけでなく、判断の筋道を見る

### 8.2 評価時に重視すること

- AIを便利な文章生成装置としてしか見ていない回答は、理解不足とみなす
- 人間レビューと責任の位置づけが書かれていれば評価しやすい
- 技術論だけでなく、運用やガバナンスまで触れている回答は高評価
