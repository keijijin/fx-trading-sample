# AI駆動開発研修テキスト スライド本文案

## 1. 目的

本資料は、`AI_Driven_Development/AI駆動開発_研修テキスト_スライド骨子.md` をもとに、各スライドへ配置する文章案をまとめたものである。  
話す原稿ではなく、スライド上に載せる見出し、本文、補助文を意識している。

## 2. スライド本文案

### Slide 1. タイトル

**タイトル**

現場事例で学ぶ AI駆動開発

**サブタイトル**

`fx-trading-sample` による設計・実装・検証の実践

**補助文**

AIを前提にした SDLC 再設計の実務

### Slide 2. 今日伝えたいこと

**見出し**

今日のメッセージ

**本文**

1. AI駆動開発は、コード生成ではなく開発様式の再設計である
2. 良い成果は、良い文脈設計と良いレビューから生まれる
3. 導入は、小さく試し、測り、改善しながら広げる

### Slide 3. AI駆動開発とは何か

**見出し**

AI駆動開発の定義

**本文**

- 生成AIとAIエージェントを SDLC 全体へ組み込む
- 要件定義、設計、実装、レビュー、運用を再設計する
- 重要判断は人間が保持する

**補助文**

Human-prompted -> Agent-executed -> Human-reviewed

### Slide 4. 関連概念の整理

**見出し**

コード補完との違い

**本文**

- AI-assisted coding: 点的支援
- Vibe coding: 速いが本番規律に弱い
- Agentic engineering: 自律実行を含む
- Spec-driven / Context Engineering: 品質安定化の基盤

### Slide 5. なぜ今か

**見出し**

AI駆動開発が求められる背景

**本文**

- 人材制約
- タイム・トゥ・マーケット圧力
- レガシーと技術的負債

**補助文**

ツール成熟、マルチエージェント化、接続基盤の整備が進んでいる

### Slide 6. 効果は一様ではない

**見出し**

「速くなる」は条件付き

**本文**

- 小さな新規実装では効きやすい
- 既存の成熟コードベースでは逆効果もある
- 最後の30%は人間の設計とレビューが重い

**補助文**

感覚ではなく、ベースラインと指標で判断する

### Slide 7. AI利用の主要リスク

**見出し**

4つのリスク軸

**本文**

- 品質
- セキュリティ
- 自律性
- コンプライアンス

**補助文**

AIは、もっともらしく間違えることがある

### Slide 8. ガバナンスの原則

**見出し**

Bounded Autonomy

**本文**

- 低リスク: 自律実行可
- 中リスク: 実行後に人間レビュー
- 高リスク: 事前承認必須

**補助文**

リスクに応じて線引きし、個人判断へ委ねすぎない

### Slide 9. Context Engineering

**見出し**

良いAI出力は、良い文脈から生まれる

**本文**

- 何を見せるか
- 何を見せないか
- どの順で渡すか
- どこで完了とするか

**補助文**

プロンプトの工夫だけでは不十分

### Slide 10. 良い依頼文の型

**見出し**

AI依頼テンプレート

**本文**

- 目的
- 背景
- 前提
- 依頼内容
- 期待する出力
- 禁止事項
- 完了条件

**補助文**

良い依頼は、良い準備から始まる

### Slide 11. 今回の題材

**見出し**

`fx-trading-sample` の3つのケース

**本文**

- CDC shadow topic 検証
- CQRS 導入条件の整理
- シャーディング・ドメイン分割ロードマップ

### Slide 12. ケース1

**見出し**

CDC shadow 検証

**本文**

- `outbox_event -> WAL -> Debezium -> Kafka Connect -> shadow topic`
- 本番切替ではなく、shadow で経路成立を確認
- 検証範囲と次段課題を分ける

**補助文**

段階導入と安全側設計の実例

### Slide 13. ケース2

**見出し**

CQRS 導入判断

**本文**

- CQRS は write path の特効薬ではない
- read model は読み取り分離に効く
- source of truth は ACID core に残す

**補助文**

AIに論点整理をさせ、採否条件は人間が決める

### Slide 14. ケース3

**見出し**

シャーディング方針

**本文**

- `accountId` ベース shard を第一候補
- shard 間 2PC / XA は禁止
- Camel ルートは配線専用

**補助文**

構造変更は、条件が揃った時だけ行う

### Slide 15. 研修で身につける力

**見出し**

受講後にできること

**本文**

- 依頼前に仕様と制約を整える
- AI出力をレビューする
- 設計判断を文書として残す
- 小規模パイロットを設計する

### Slide 16. カリキュラム

**見出し**

標準2日版の全体像

**本文**

- Day 1: 基礎理解、リスク、依頼文設計
- Day 2: ケース演習、導入計画、達成度診断

**補助文**

1日版、半日版にも圧縮可能

### Slide 17. 達成度診断

**見出し**

知識ではなく、実践力を測る

**本文**

- 事前診断
- 演習評価
- 事後診断
- 5観点評価: 理解 / 設計 / 実行 / レビュー / 展開

### Slide 18. 導入効果の測り方

**見出し**

速度だけで測らない

**本文**

- 初稿作成時間
- レビュー差戻し率
- 手戻り件数
- 品質指標
- 開発者満足度

**補助文**

速さと引き換えに、品質やレビュー負荷を落とさない

### Slide 19. 導入の進め方

**見出し**

小さく始めて標準化する

**本文**

1. 低リスク業務を選ぶ
2. テンプレートを用意する
3. レビュー責任を決める
4. 指標を取る
5. 成功した型だけを標準化する

### Slide 20. まとめ

**見出し**

AI時代の開発責任を再設計する

**本文**

- AI駆動開発は、便利ツールの話ではない
- 人間の判断、レビュー、説明責任はむしろ重要になる
- 実践は、現場文脈に合わせて段階導入すべきである
