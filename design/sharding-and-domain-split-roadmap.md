# シャーディング・ドメイン分割ロードマップ

本 PoC の **単一約定コア（fx-core）＋ ACID 約定** は、ある時点のスループット上限を持つ。負荷試験で **HTTP / DB / Outbox / Kafka** を底上げしても頭打ちになる場合、要件がその上限を超えるなら **アーキテクチャ変更**が必要になる。

## いつ検討するか

- スパイク試験（`loadtest/run_spike_scale_test.py`）で **spike 相**に Outbox 滞留・Kafka lag が集中し、レプリカを増やしても改善が頭打ちになる。
- `loadtest/verify_db_connection_budget.py` で **PostgreSQL `max_connections`** に対し JDBC 積み上げが逼迫する（レプリカ増でしか伸ばせないが、DB 接続が先に限界）。
- ビジネス要件として **秒間約定件数**または**口座あたり更新頻度**が、現行の単一シャードでは物理的に足りないと判断されたとき。

## 方向性（概念）

| 施策 | 概要 | トレードオフ |
|------|------|----------------|
| **口座・顧客シャード** | シャードキー（例: `accountId` のハッシュ）ごとに約定 DB / fx-core デプロイを分離 | クロスシャード取引・集約・レポートが複雑化。マイグレーションと運用コスト。 |
| **約定パスと参照系の分離** | 約定は極小トランザクションに限定し、参照・集計は CQRS / 読みモデルへ | 整合性ラグの説明・実装が必要。 |
| **イベント駆動の後続のみスケール** | 現状どおり Saga / Kafka はスケールしやすいが、**ボトルネックが約定コア側**なら根本解決にはならない | 約定コアが限界のときはシャード等が必要。 |

## 実装に向けたフェーズ（目安）

1. **計測の固定化**  
   相別メトリクス（Outbox・lag・Hikari）と接続数バジェットで、限界が **約定**か **後続**かを切り分ける。
2. **シャード設計**  
   キー選定、データ配分、シャード間禁止事項（同一取引内で複数シャードに触れない等）、障害時の扱い。
3. **移行**  
   口座の再配置、二重書き、読み切り替え、ロールバック手順。
