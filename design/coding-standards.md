# FXトレーディングシステム コーディング基準

## 1. 目的

本基準は、`design/design.md` に定義された FX トレーディングシステムのアーキテクチャを、実装時にぶらさず維持するための共通ルールである。対象は PoC を含む全サービスとし、特に `高パフォーマンス`、`高信頼性`、`整合性維持` を優先する。

## 2. 基本原則

1. 約定コアは単一サービス内の ACID トランザクションで閉じる。
2. 後続処理は Kafka イベント駆動の Saga で連携し、最終的整合を採用する。
3. イベント送信保証は Outbox で実現し、Consumer 側で冪等を担保する。
4. 補償は DB ロールバックではなく、業務的な逆取引・取消で実装する。
5. Camel は配線に徹し、業務判断・状態遷移・補償判定は Service 層に集約する。
6. 連携処理は独自実装より先に Camel 4 の EIP / Component で表現する。

## 3. レイヤ責務

### 3.1 Camel Route

- Kafka / Timer / SQL / direct エンドポイントの接続に限定する。
- ヘッダ設定、イベント種別分岐、Bean 呼び出し、再送入口を担当する。
- ルーティング、分岐、集約、冪等、例外処理、再送は、まず Camel 4 標準の EIP / Component で実現する。
- 約定判定、残高計算、補償範囲判定のような業務ロジックを書かない。
- `choice`、`split`、`aggregate`、`filter`、`idempotentConsumer`、`onException`、`deadLetterChannel`、`transacted`、`wireTap` などで置き換えられる手書き制御は追加しない。

### 3.2 Service

- 業務判定、トランザクション境界、状態遷移、補償判定を担当する。
- DB 更新は必ず Service から始める。
- 例外は握りつぶさず、致命 / 非致命を明確に分ける。

### 3.3 Repository / DAO

- 永続化処理に専念する。
- 業務ルールを持ち込まない。
- 排他制御や一意制約の前提を崩す更新を行わない。

## 4. トランザクションと整合性

- FX Core Service では、約定前コンプライアンス、二重約定防止、約定登録、残高拘束、建玉更新、`trade_saga` 初期登録、`outbox_event` 登録までを 1 トランザクションで処理する。
- 後続サービスでは、サービス内ローカル更新のみをローカルトランザクションとして扱い、他サービスへの整合性はイベントで接続する。
- 後続処理のために同期 RPC を追加しない。依存関係はイベント順序で表現する。
- Camel で扱うトランザクション境界や再送制御は、可能な限り `transacted`、`onException`、コンポーネント設定など標準機能で表現する。

## 5. イベント設計

- すべてのイベントに `eventId`、`correlationId`、`eventType`、`messageKey`、`sourceService` を持たせる。
- 事実通知は過去形イベント名、指示は `Requested` 接尾辞で統一する。
- 補償要求イベントは `fx-compensation-events` に集約する。
- 新しい Saga ステップを追加する場合は、成功イベント、失敗イベント、補償要求、補償完了、Saga 状態、テスト観点を同時に定義する。
- Kafka 送受信、DB ポーリング、再送ルート、死信経路は Camel Component と EIP の組み合わせで表現し、独自フレームワークを増やさない。

## 6. 冪等・排他

- Consumer は `processed_message` などで `eventId` 単位の冪等制御を必ず行う。
- 同一 `order_id` の二重約定を防ぐため、楽観ロックと一意制約を前提に実装する。
- `version_no` を持つエンティティは更新時に必ず整合性チェックを行う。
- `SELECT ... FOR UPDATE` は本当に必要な箇所に限定し、ロック時間を短く保つ。

## 7. 金額・状態・エラー

- 金額計算は `BigDecimal` を使用し、`double` / `float` を使わない。
- 状態値は文字列の直書きではなく、列挙型または定数で管理する。
- `TradeNotificationFailed` のような非致命失敗では取引本体を巻き戻さない。
- 例外ログには最低でも `tradeId`、`orderId`、`correlationId`、`eventId` を含める。

## 8. テスト基準

- 正常系として、約定から Saga 完了までの一連のフローを検証する。
- 異常系として、カバー失敗、会計失敗、受渡失敗時の補償範囲を検証する。
- 冪等として、同一 `eventId` 再受信で二重反映しないことを検証する。
- 排他として、同一注文の二重実行防止を検証する。
- Outbox と再送として、送信失敗時の `RETRY` と上限超過時の異常状態を検証する。

## 9. 変更時の必須確認

- アーキテクチャ変更時は `design/design.md` とこの基準書の両方を更新する。
- 新規イベント追加時は、イベント経路マトリクスとテスト観点を更新する。
- 新規サービス追加時は、Camel の責務境界と補償方針を明文化する。
- 新しい連携ロジックを追加する際は、Camel 4 の EIP / Component で代替できない理由がある場合だけ独自制御を採用する。
